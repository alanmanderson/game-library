# Backgammon AI: Executive Report

## Project Summary

We built a neural network-based backgammon AI to replace the random-move bot player in the Backgammon Online application. The model was trained from scratch using self-play data generated by the existing game engine, following the methodology pioneered by TD-Gammon (Tesauro, 1992).

### Key Results

| Matchup | Win Rate | Games Tested |
|---------|----------|-------------|
| **Neural AI vs Random** | **98.75%** | 400 |
| **Neural AI vs Heuristic** | **73.25%** | 400 |
| Random vs Heuristic (baseline) | 1.5% | 200 |

The trained model crushes the previous random bot (99% win rate) and convincingly defeats a hand-crafted heuristic opponent that itself wins 98.5% against random. The model has been integrated into `bot_service.py` and is now the default bot opponent.

## 1. Architecture

### Neural Network Design

The model follows the TD-Gammon architecture, the gold standard for backgammon AI since 1992:

```
Input Layer:   198 features (standard Tesauro encoding)
Hidden Layer 1: 80 units, sigmoid activation
Hidden Layer 2: 80 units, sigmoid activation
Output Layer:   5 units, sigmoid activation
Total Parameters: 22,805
```

### Input Encoding (198 Features)

The board position is encoded using the standard 198-feature scheme:

- **192 features**: 24 board points x 4 units x 2 players
  - Truncated unary encoding per point: 0 checkers = [0,0,0,0], 1 = [1,0,0,0], 2 = [1,1,0,0], 3 = [1,1,1,0], n >= 4 = [1,1,1,(n-3)/2]
- **2 features**: Bar checkers (one per player, normalized by /2)
- **2 features**: Borne-off checkers (one per player, normalized by /15)
- **2 features**: Turn indicator ([1,0] or [0,1])

This encoding is always computed from the current player's perspective, enabling the network to learn a single evaluation function that works for both sides.

### Output Interpretation

The 5 output units represent:

| Output | Meaning |
|--------|---------|
| `output[0]` | P(current player wins) |
| `output[1]` | P(current player wins gammon) |
| `output[2]` | P(current player loses gammon) |
| `output[3]` | P(current player wins backgammon) |
| `output[4]` | P(current player loses backgammon) |

**Equity calculation**: `E = 2*P(win) - 1 + P(win_gammon) - P(lose_gammon) + P(win_bg) - P(lose_bg)`

## 2. Training Methodology

### Two-Phase Training Pipeline

We developed an optimized two-phase training approach that combines supervised pre-training with TD reinforcement learning:

#### Phase 1: Self-Play Data Generation (50,000 games)
- Played 50,000 complete games using random move selection
- Recorded board positions at 30% sampling rate (to manage dataset size)
- Generated **1,337,979 training positions** with game outcomes
- Processing speed: 8.7 games/second
- Duration: ~95 minutes

#### Phase 2: Supervised Pre-Training (30 epochs)
- Trained the network to predict game outcomes from board positions
- Used MSE loss with Adam optimizer (lr=0.001)
- Batch size: 512, shuffled per epoch
- Loss decreased from 0.1086 to 0.1001 over 30 epochs
- Post-training evaluation: **99.0% win rate vs random**
- Duration: ~31 minutes

#### Phase 3: TD Self-Play Refinement (5,000 games)
- Used the pre-trained model for both sides in self-play
- Applied batch TD(lambda=0.7) updates after each game
- Adam optimizer with lr=0.001, gradient clipping at 1.0
- 15% initial exploration rate, decaying at 0.9999 per game
- Average game length: 74-87 turns (varies with model strength)
- Duration: ~6 hours at 0.2 games/second

### Why This Approach?

Pure TD self-play (the original TD-Gammon method) was too slow at 0.1 games/second because every move requires evaluating all candidate positions. Our two-phase approach leverages fast random game generation (~9 games/sec) to create a strong initial model, then refines it through slower but higher-quality TD self-play.

## 5. Move Selection Algorithm

At each decision point, the bot:

1. Gets all valid moves from the game engine
2. For each candidate move:
   a. Saves the board state (snapshot)
   b. Applies the move internally
   c. Encodes the resulting position as 198 features
   d. Runs the neural network forward pass
   e. Computes the position equity
   f. Restores the board state
3. Selects the move with the highest equity

For doubling cube decisions:
- **Accept double**: If equity > -0.5 (dead cube take point)
- **Offer double**: If equity > 0.5

## 6. Integration

The model has been integrated into `backend/app/services/bot_service.py` with the following design:

- **Lazy loading**: The model is loaded on first use, not at server startup
- **Graceful fallback**: If the model file is missing or loading fails, the bot falls back to random moves
- **No new dependencies in production**: PyTorch is only required on the ML training machine. The model can optionally be exported to ONNX format for lighter inference.

### How to Use

```python
# The model is automatically used by the bot service
# No code changes needed - it loads from ml/models/backgammon_model_final.pt

# For manual usage:
from ml.bot_integration import MLBotPlayer
bot = MLBotPlayer("ml/models/backgammon_model_final.pt")

# Select best move
move = bot.select_move(engine)

# Evaluate position
analysis = bot.get_position_analysis(engine)
# Returns: {win_probability, equity, gammon_probabilities, ...}

# Doubling decisions
should_take = bot.should_accept_double(engine)
should_offer = bot.should_double(engine)
```

## 9. Limitations and Future Work

### Current Limitations
1. **No multi-ply search**: The model evaluates positions at 0-ply (single position evaluation). Adding 1-ply or 2-ply lookahead (averaging over all 21 dice combinations) would significantly improve play.
2. **No hand-crafted features**: The 198-feature encoding captures raw board state but misses strategic concepts like prime strength, blot vulnerability, and race metrics that boosted TD-Gammon v2.1.
3. **CPU-only training**: Training was limited by CPU speed. GPU training would enable 10-100x more games.
4. **Single network**: GNU Backgammon uses separate networks for contact, race, and crashed positions.

### Recommended Next Steps
1. **Add 1-ply lookahead**: For each candidate move, evaluate the resulting position across all 21 dice outcomes. This alone could improve win rate by 5-10%.
2. **Continue TD training**: The model would benefit from 100K+ additional TD self-play games with the pre-trained weights.
3. **Add expert features**: Expanding the input encoding to ~250 features (adding prime detection, pip count, blot exposure) would improve evaluation accuracy.
4. **Export to ONNX**: For production deployment, export the model to ONNX format to eliminate the PyTorch dependency.

## 10. Methodology: How Sub-Agents Were Leveraged

This project was built by coordinating multiple specialized agents:

1. **Codebase Explorer**: Analyzed the existing game engine (`game_engine.py`, `bot_service.py`, `models.py`, `schemas.py`) to understand board representation, move format, and integration points.

2. **Data Research Agent**: Searched the internet for backgammon datasets, finding resources including GNU Backgammon databases, FIBS game records, Hardy Huebener's match archive (120+ tournament matches), and the bgmoves dataset (39,690 analyzed positions). Ultimately determined that self-play data generation was the optimal approach.

3. **ML Architecture Researcher**: Investigated TD-Gammon's 198-input encoding, network architecture (hidden layer sizes, activation functions), training methodology (TD-lambda, self-play), and modern approaches (GNU Backgammon's multiple networks, WildBG's ONNX-based approach). This informed our architecture decisions.

4. **Data Engineer**: Designed the data pipeline to efficiently generate 1.34M training positions from 50,000 self-play games, with strategic position sampling and proper encoding.

5. **Move Validator / Expert Judge**: Implemented in `move_validator.py` - scores moves using established backgammon principles (bearing off, hitting, making points, avoiding blots, building primes) and validates that all model-selected moves are legal.

6. **Training Orchestrator**: Managed the two-phase training pipeline, monitoring convergence and adjusting hyperparameters.

7. **Evaluation Agent**: Ran comprehensive evaluation matches (400+ games per matchup) against random and heuristic baselines, producing detailed statistics.
